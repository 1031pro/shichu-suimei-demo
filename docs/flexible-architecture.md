# 柔軟化アーキテクチャ

## 基本思想

四柱推命アプリは、流派差分をコードに直書きすると保守が難しくなる。

そのため、以下を分離する。

- 共通計算エンジン
- 流派ルール設定
- 表示UI
- 鑑定文テンプレート
- PDF・納品物出力

## レイヤー構成

### 1. Core Engine

どの流派でも比較的共通しやすい計算を担当する。

- 干支計算
- 四柱算出
- 天干・地支
- 五行・陰陽
- 通変星
- 十二運
- 空亡
- 大運
- 年運

### 2. Rule Profile

流派差分や鑑定者ごとの設定を担当する。

設定化候補:

- 節入り時刻の扱い
- 早子時・夜子時
- 真太陽時補正
- 蔵干表
- 大運起算方法
- 大運の順行・逆行判定
- 空亡算出方式
- 表示する星・項目
- 使わない項目

### 3. Interpretation Layer

鑑定文や解釈文を扱う。

ライトプランでは未使用。

将来的な追加候補:

- 通変星ごとの解釈文
- 十二運ごとの解釈文
- 大運ごとの解釈文
- 年運ごとの解釈文
- 鑑定者独自のテンプレート
- AI生成プロンプト

### 4. Presentation Layer

表示・納品形式を担当する。

- Web画面
- スマホ画面
- 印刷用画面
- PDF鑑定書
- 顧客向け共有ページ

## データ設計イメージ

流派設定はJSONまたはDBで管理する。

例:

```json
{
  "profileId": "default",
  "name": "標準設定",
  "useTrueSolarTime": false,
  "midnightRule": "standard",
  "hiddenStemTable": "default",
  "luckCycleStartMethod": "standard",
  "showItems": {
    "hiddenStems": true,
    "tenGods": true,
    "twelveFortunes": true,
    "void": true,
    "majorLuck": true,
    "annualLuck": true
  }
}
```

## 実装方針

最初は1つの標準ルールでMVPを作る。

ただし、後から流派差分を追加できるように、計算関数の中に固定値を散らさない。

将来的には、顧客ごとに `ruleProfile` を切り替えることで、同じアプリ基盤から複数案件に対応できるようにする。

## 独自性の扱い

既存アプリを調査するときは、独自部分を以下に分類する。

- 単なる実装上の癖
- 特定流派に由来するルール
- 鑑定者本人の思想・鑑定手順
- 商品として差別化できる独自機能

価値があるのは、鑑定者本人の思想や鑑定手順が反映された部分。
